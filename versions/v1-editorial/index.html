Migration story

The homepage should teach the product in one path.

The value is not just “deploy”. The value is “start on shared infrastructure, then move to your own machine when the product deserves it.”

  1. 01

    Create workspace

    Start with the product shell, not a pasted token.

  2. 02

    Import repository

  3. 03

    Deploy to shared runtime

    Use the managed path first, with operations and audit visible from the start.

  4. 04

    Attach a VPS

    Create a node key, copy the join command, and wait for runtime heartbeat.

  5. 05

    Migrate the app

    Move the workload onto your own runtime without changing the app model.

Object model

Five objects explain the whole system.

Fugue becomes easier to trust once the model is visible: a workspace maps to a tenant, projects hold apps, runtimes receive deploys, and operations record every change.

  • Workspace maps to tenant
  • Apps deploy onto a chosen runtime
  • Operations are first-class, not hidden spinners
  • Audit trail stays close to every action

curl -sS "${FUGUE_BASE_URL}/healthz"

curl -sS "${FUGUE_BASE_URL}/v1/apps/import-github" \
  -H "Authorization: Bearer <tenant-api-key>" \
  -H "Content-Type: application/json" \
  -d '{
    "repo_url":"https://github.com/example/static-site",
    "runtime_id":"runtime_managed_shared"
