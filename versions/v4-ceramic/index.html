Shared-first deployment control plane

From GitHub to a live app, then onto your own VPS.

Fugue-web is the product wrapper around the already deployed Fugue core. It gives a new visitor one clean mental model: import a repository, run on shared k3s, inspect the whole operation trail, and later move the same app onto a machine you attach.

Route logic

The product story should feel like a control object, not a slogan wall.

These three raised panels carry the entire pitch: take code from GitHub, land on a managed shared runtime, and leave cleanly when a dedicated machine becomes worth it.

System objects

Five nouns keep the control plane legible.

Fugue-web will eventually need landing, auth, docs, and console layers. These objects define the shared product grammar so every future surface stays consistent.

Object map

Workspace, project, app, runtime, and operation stay visibly distinct.

That keeps the product layer from collapsing into a generic dashboard where every action, resource, and result gets flattened into one noisy surface.

workspace Tenant boundary

Maps the account shell and defines the multi-tenant boundary.

project Default project

Groups imported apps so the product surface does not collapse into one bucket.

app Imported app identity

The same app survives repository intake, shared deploy, and later migration.

runtime Shared or attached machine

Can point at managed shared infrastructure or a VPS you attach later.

operation Build, deploy, migrate

Records execution history so shipping work stays inspectable over time.

Quickstart

Inspect the live product surface from the public API.

export FUGUE_BASE_URL="https://api.fugue.pro"

curl -sS "${FUGUE_BASE_URL}/healthz"

curl -sS "${FUGUE_BASE_URL}/v1/apps/import-github" \
  -H "Authorization: Bearer <tenant-api-key>" \
  -H "Content-Type: application/json" \
  -d '{
    "repo_url":"https://github.com/example/static-site",
    "runtime_id":"runtime_managed_shared"
  }'
