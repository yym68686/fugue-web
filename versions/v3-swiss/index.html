Product line

Fugue is not hiding infrastructure. It edits the route through it, so teams can start on shared k3s now and leave onto their own runtime later.

migration route / github -> shared runtime -> user vps
static preview / auth shell planned at /auth/sign-in

Primary statement

The product is the route, not the screenshot.

A cold visitor should understand Fugue in one pass: import a repository, run it on shared infrastructure, then move the same app onto an attached machine later.

  • repo intake / public GitHub
  • first deploy / runtime_managed_shared
  • exit path / attach node and migrate app

Operational bands

Three bands explain the whole surface.

01

Import a public GitHub repository

Point Fugue at a repository, choose a branch and build mode, and let the control plane prepare the first deploy.

repo_url / github.com/example/static-site branch / main builder / auto
02

Deploy on shared k3s first

Use the managed shared runtime as the fastest route to a public URL while build logs, deploy operations, and audit stay visible.

runtime_id / runtime_managed_shared operation / build, deploy, route visibility / logs + audit trail
03

Attach a VPS and migrate later

Issue a node key, join a machine, confirm heartbeat, and move the same app onto your own runtime without changing the control model.

node key / issue once heartbeat / runtime stable migrate / shared -> attached

System objects

Five objects explain the control plane.

workspace Tenant boundary
project Default project
app Imported app
runtime Shared or attached
operation Deploy, scale, migrate
Workspace

Maps to the tenant shell and defines the multi-tenant boundary.

Project

Groups imported apps so the product surface does not flatten into one bucket.

App

The same app identity survives import, shared deploy, and later migration.

Runtime

Can point at managed shared infrastructure or a machine you attach later.

Operation

Records build, deploy, and migrate work as first-class execution history.

export FUGUE_BASE_URL="https://api.fugue.pro"

curl -sS "${FUGUE_BASE_URL}/healthz"

  -H "Authorization: Bearer <tenant-api-key>" \
  -H "Content-Type: application/json" \
  -d '{
    "repo_url":"https://github.com/example/static-site",
    "runtime_id":"runtime_managed_shared"
  }'

Questions before trust

Handle objections without flattening the product.

Answer / 01

Not in the generic sense.

Fugue is a control plane with a hosted shared runtime path first, then a path toward your own attached VPS runtime.

Answer / 02

No, that is exactly what the shared route removes.

The landing page leads with the shared runtime because that is the shortest route to a running app. Your own VPS becomes relevant when you want that control later.

Answer / 03

Yes, that migration path is the core product story.

Start on shared infrastructure, attach a VPS with a node key, then migrate the same app through the same control surface.

Answer / 04

Not in the surface shown here.

The current landing story keeps the product boundary explicit: the available import flow in this public surface is for public GitHub repositories.

Answer / 05

Because the product shell still matters.

This preview focuses on the public marketing surface today, while auth and onboarding remain the next product layer to implement.
